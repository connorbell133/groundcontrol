Grounding Context

Codebase Context. agent-runner is a personal, tailnet-only mobile-first PWA (Hono/TS backend, vanilla JS frontend, ~850 LOC) on the owner's machine. It browses folders, launches claude remote-control sessions via node-pty — optionally in git worktrees off a chosen branch, inheriting repo context from the nearest parent .git — shows pairing QR codes, lists/kills sessions, and journals every lifecycle event to data/journal.json. Sessions live in an in-memory Map (lost on runner restart); the UI polls every 2.5s. Evidence dossiers were extracted per axis; key findings: session timestamps ship to the client but are never rendered; the journal holds every past run's exact config but no route exposes it; logs are a one-shot fetch behind a collapsed disclosure; the manifest has no shortcuts/share-target and the palette has no dark mode. External research found heavy convergence across comparable tools (muxplex, amux, Nimbalyst, ccmanager, Dozzle, Opcode, Anthropic's Agent View) on: status-aware session lists, completion/blocked notifications, one-tap relaunch, run history, last-output previews, and auto dark mode. A fresh-context verifier spot-checked every basis against the repo and iOS platform reality; two candidates died there.

1

Journal-powered run history: recents, one-tap relaunch, resurrection

relaunch-loopsConfidence 90%Complexity Medium5-frame convergence
Description
One new endpoint — GET /api/journal/recent, returning the last N distinct launch configs from session.start events — powers three surfaces: a recents strip atop the Browse tab (tap to pre-fill the launch sheet, long-press to launch), a Relaunch button on exited/error session cards, and "lost — runner restarted" headstone cards for journal starts with no matching exit, each with one-tap resurrection.
Basis
direct: src/sessions.ts journals the full config (folder, branch, spawnMode, permissionMode) on every start; src/server.ts has no journal route; the verifier confirmed real orphaned starts in data/journal.json. external: one-tap resume converges across ccmanager, muxplex, and Opcode.
Rationale
The evidence dossier documents an 8-step morning re-entry sequence where steps 3–6 are pure navigation overhead. The data to eliminate it has been written to disk on every run since day one — this is a display problem, not a data problem. It also converts the app's worst moment (runner restarted, sessions vanished, UI pretends they never existed) into a two-tap recovery.
Downsides
The journal is append-only JSON that grows unbounded — the endpoint should read the tail, and eventually the journal wants rotation. Headstone detection needs care not to resurrect intentionally-killed sessions.
2

Live session cards: activity age + last-line ticker

session-awarenessConfidence 85%Complexity Low5-frame convergence
Description
Stamp lastOutputAt in the PTY data handler and add a lastLine field (last non-empty log line, ~100 chars) to the session payload. Cards then show an activity readout — "output 3s ago" vs "quiet 12m" — an "up 47m" age chip computed from the startedAt that already ships to the client but is never rendered, a live elapsed counter during starting, and a faint monospace ticker of the agent's most recent output. Error dying-words land on the card face instead of hiding behind a collapsed disclosure.
Basis
direct: verifier confirmed publicView strips only proc and log, so startedAt reaches the client unrendered; the 400-chunk ring buffer already holds the last line; dossier pain point #1: "once ready, sessions are indistinguishable." external: last-output previews on tiles in Dozzle and Anthropic's Agent View; "last seen" timers are the Home Assistant pattern.
Rationale
The core phone question is "is it doing something?" — today that costs a tap per session per check. This converts the Sessions tab from a launcher receipt into a monitoring surface for one server field and a few card lines.
Downsides
The card render loop diffs on state/pairingUrl only and rewrites innerHTML — the ticker must join the diff key without nuking open log disclosures (same fix idea 7 needs). Raw PTY last-lines can be noisy; needs light filtering.
3

ntfy tee on journal events

notificationsConfidence 90%Complexity Low
Description
The journal() function is the single choke point every lifecycle event already passes through. Add an env-gated, fire-and-forget POST to an ntfy topic on session.ready (with the pairing URL as the notification's click action) and on error/nonzero exit. The phone notification becomes the pairing handoff: launch, pocket the phone, get tapped on the shoulder when the QR is ready or the run dies.
Basis
direct: requirements R20 specifies ntfy and the owner's infra already runs it; evidence-notifications.md confirms journal events forward nowhere today. external: completion/blocked push is the single strongest convergence signal across amux, Nimbalyst, and Dozzle.
Rationale
The 2.5s poll only works while staring at the app; the actual usage pattern — launch and pocket — is exactly what polling can't serve. This is the deferred roadmap item shrunk to its nuance-sized core: ~15 lines, no Web Push plumbing, no service-worker changes.
Downsides
Requires the ntfy app (or browser subscription) on the phone; topic name is a shared secret to keep private on a public ntfy server, or self-host.
4

Safelight dark mode

pwa-ergonomicsConfidence 85%Complexity Medium
Description
A prefers-color-scheme: dark token block over the already-fully-tokenized palette — dark paper ground, bone text, vermillion and stamp-green dimmed to safelight intensity — plus a matching theme-color swap. The darkroom-safelight framing: the canonical use is checking overnight agents from bed, and today that's a full-brightness warm-paper flashbang.
Basis
direct: styles.css is a single :root token block with no prefers-color-scheme anywhere; theme-color is hardcoded. iOS 13+ support confirmed by the verifier. external: every comparable tool (muxplex, Dozzle, amux) ships auto dark/light.
Rationale
Because the palette is fully tokenized, this is an afternoon of picking hex values — and it fixes the single most physically uncomfortable moment in the app's daily loop, the 2am "did it finish?" check.
Downsides
Every tinted state (chips, pills, QR panel, provision shimmer) needs a contrast re-check in the dark block; the server-rendered QR SVG colors also need a dark-aware variant or a neutral treatment.
5

Intent line: name the run at launch

session-awarenessConfidence 80%Complexity Low
Description
The server already accepts a session name and falls back to folder-a1b2; the launch sheet has the field but frames it as an afterthought ("auto-generated"). Reframe it as the intent line — "what's this run for?" — so every downstream surface inherits task identity: cards read "fix checkout race" instead of "ehugs-9c1b", recents become scannable, ntfy messages say what finished, and the journal becomes a legible work log.
Basis
direct: name plumbing exists end-to-end (verifier confirmed); the gap is purely UX framing in the sheet. external: redock.dev's mobile-tmux insight — session naming tied to task identity is what makes multi-session lists legible after a phone disconnect.
Rationale
Cheapest first domino: it multiplies the value of ideas 1–3 (history entries, tickers, and notifications all become meaningful) for one input's worth of copy and emphasis.
Downsides
Optional fields get skipped; the fallback name should stay good enough that skipping costs little.
6

Wake-sync on return-to-app + staleness readout

pwa-ergonomicsConfidence 85%Complexity Low
Description
iOS freezes PWA timers in the background, so every unlock shows data as old as your pocket time until the throttled intervals recover. Add a visibilitychange listener that fires refreshSessions() and health() immediately on return-to-foreground, plus a small "synced 3s ago" readout beside the health LED so stale data is visibly stale rather than confidently wrong.
Basis
reasoned: the only refresh drivers are two setInterval calls (verifier confirmed no visibilitychange listener exists); iOS background-timer throttling is documented WebKit behavior. The unlock moment is both the stalest frame and the one where the user most wants truth.
Rationale
For a launch-and-pocket tool, the return-to-app moment is the product. One event listener and one timestamp span make it trustworthy.
Downsides
None material; the readout costs a few pixels of header space.
7

Log copy/share + render-preserving live tail

log-accessConfidence 65%Complexity Medium
Description
Two buttons on the log disclosure row — copy (navigator.clipboard) and share (navigator.share, opening the iOS share sheet) — plus a live tail: while a log is open and its session live, re-fetch on the existing 2.5s tick with stick-to-bottom scroll. Requires a render fix first: the card renderer rewrites innerHTML on state change, which would destroy an open log — refresh the log node independently of the card template.
Basis
direct: the toggle handler fetches exactly once; log text is trapped in a 220px pre with no way out — finger-selection of monospace text is broken on phones. navigator.share for text is supported on iOS Safari. The verifier flagged the innerHTML re-render conflict; kept with the redesign note because the fix is modest and known.
Rationale
When an agent errors, the next action is always "get this text somewhere else" — into Claude, a note, a message. Today that means screenshotting partial output. Copy/share is ~5 lines; the tail turns the log from a snapshot into a stream using polling that already ticks.
Downsides
Lower confidence than the rest: the live-tail mechanism as originally proposed was refuted and needs the render redesign; copy/share alone is trivially safe and could ship first.

Rejection Summary

#IdeaReason Rejected
1Manifest shortcuts (long-press home icon)Refuted by verification: iOS Safari does not implement Web App Manifest shortcuts — zero value on the owner's iPhone; Android/desktop only.
2Haptic grammar on state transitionsRefuted by verification: navigator.vibrate is not supported in iOS Safari/WebKit at all.
3OS icon badge via setAppBadgeWeakened: iOS requires Notification permission before badging works, killing the "two lines" pitch; away-from-app signaling is better served by the ntfy tee (survivor 3).
4Expected-duration baseline ("usually ~9s here")Journal too sparse for per-folder medians today, and real provision time is ~2s — the framing was wrong. Revisit when the journal grows.
5Rejoin instead of duplicate-name errorSound but low-frequency for a solo user; below the value bar in a seven-slot list.
6Session-grid output thumbnails (muxplex-style)Duplicates a stronger, cheaper idea — the last-line ticker (survivor 2) delivers the same glanceability without ANSI snapshot rendering.
